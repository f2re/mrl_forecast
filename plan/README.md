# План модернизации MRL Forecast

## Цель

Довести проект от демонстрационного прототипа до проверяемой системы радарного
наукастинга. Главный критерий: система не должна выдавать визуально убедительный
прогноз, если источник, геометрия, временная шкала или модель не прошли контроль
качества.

## Зафиксированные проблемы

- Радарная карта переворачивается по оси север-юг.
- Географическая привязка строится через Web Mercator без корректного локального
  преобразования расстояний.
- Архивная подготовка и оперативный инференс используют разные алгоритмы
  гриддинга.
- При ошибках декодирования или источника operational-путь может незаметно
  подставить синтетические данные.
- AWS-источник существует и отвечает, но live-путь зависит от локальной
  конфигурации региона boto3.
- Реальный шаг сканов KOKX не равен жестко заданным 15 минутам.
- Train/validation split допускает утечку почти одинаковых окон между выборками.
- Модель из реестра выдает почти однородное поле отражаемости 24-28 dBZ.
- Графики обучения появляются только после штатного завершения процесса.
- Текущий тестовый контур проверяет только форму выхода ConvLSTM.

Подробные доказательства собраны в [00-audit-baseline.md](00-audit-baseline.md).

## Приоритеты

### P0: доверие к данным

Выполнить этапы 0-4 до любого усложнения нейросети. Пока неизвестно, что именно
попадает в модель и как оно нанесено на карту, улучшение архитектуры не имеет
инженерного смысла.

### P1: качество прогноза

Выполнить этапы 5-7. Модель допускается к использованию только после сравнения с
persistence и advection baseline на независимом тестовом периоде.

### P2: эксплуатация и новые источники

Выполнить этапы 8-9. Контур российских ДМРЛ должен развиваться отдельно от NOAA
NEXRAD, пока не определены формат, лицензия и реальные образцы данных.

## Общие правила реализации

- Не смешивать demo-данные и наблюдения ни в одной директории, выборке или API.
- Не менять ML-модель до фиксации единого preprocessing pipeline.
- Каждый этап закрывать тестами и контрольным отчетом.
- Любую временную метку хранить в UTC с timezone information.
- Любой датасет и чекпоинт сопровождать provenance metadata.
- Любую новую метрику считать отдельно по lead time.
- Не считать MSE достаточным критерием качества осадков.

## Definition of Done проекта

- Ошибка источника не превращается в прогноз.
- Offline и online обработка одного файла дают одинаковый массив.
- География проверена на контрольных радарных кадрах.
- Датасеты имеют QC-отчет, временную шкалу и независимые split.
- Модель превосходит persistence baseline на независимом test-наборе по
  согласованным метрикам.
- Реестр не предлагает незавершенные или невалидные модели.
- Графики и метрики доступны во время и после обучения.
- NetCDF содержит CRS, координаты, UTC и provenance.
- Автоматические тесты покрывают источник, обработку, модель, карту, API и экспорт.
