# Этап 2. Единый радарный preprocessing pipeline

## Проблема

Архивный dataset builder и оперативный inference преобразуют радары разными
способами. Модель обучается на одном распределении сеток, а получает другое.

## Цель

Один и тот же сырой файл должен давать одинаковый массив независимо от пути:
архив, live AWS, FTP, CLI, Flask API или тест.

## Решения, которые нужно зафиксировать до реализации

1. Какой метеорологический продукт используется:
   - lowest-elevation PPI для быстрого первого этапа;
   - либо CAPPI на фиксированной высоте, например `1 км AGL`.
2. Размер области:
   - рекомендуемый старт: радиус `250 км`;
   - сетка `256 x 256`;
   - локальная метрическая система координат.
3. Диапазон отражаемости:
   - сохранять исходные значения и mask;
   - нормализацию выполнять отдельным шагом для ML;
   - не смешивать `no data`, отрицательные dBZ и нулевую отражаемость.
4. Временной шаг:
   - рекомендуемый старт: регулярная сетка `10 минут`;
   - resampling с допустимым отклонением;
   - явная маркировка пропусков.

## Задачи

1. Создать модуль единого препроцессинга, например `src/radar_pipeline.py`.
2. Вынести декодирование, выбор поля reflectivity, sweep selection, QC,
   projection и gridding в именованные этапы.
3. Использовать pipeline из `make_dataset.py`, `adapters.py` и CLI inference.
4. Хранить параметры pipeline в metadata датасета.
5. Добавить версию pipeline, чтобы модель знала, на каких данных обучалась.
6. При загрузке модели проверять совместимость pipeline version.
7. Определить правила обработки masked array и NaN.
8. Добавить диагностику coverage и распределения dBZ.

## Предлагаемая структура metadata

```json
{
  "pipeline_version": "radar-grid-v1",
  "product": "lowest_elevation_reflectivity",
  "units": "dBZ",
  "grid": {
    "width": 256,
    "height": 256,
    "radius_km": 250,
    "crs": "local_aeqd"
  },
  "time_step_minutes": 10
}
```

## Тесты

- Один fixture-файл, обработанный archive и live-путями, дает одинаковый массив.
- Mask не превращается незаметно в осадки.
- NaN не проходят в обучающую выборку.
- Неизвестное поле reflectivity дает понятную ошибку.
- Pipeline version сохраняется в dataset metadata и checkpoint.

## Риски

- Переход на единый pipeline сделает старые датасеты и модели несовместимыми.
  Их следует пометить legacy, а не пытаться смешивать с новыми.
- CAPPI потребует больше вычислений и более строгой работы с объемными сканами.

## Критерий завершения

Для контрольного сырого файла архивная обработка, live inference и тестовый
вызов дают численно одинаковый grid и одинаковый QC-отчет.
