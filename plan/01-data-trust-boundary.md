# Этап 1. Граница доверия к данным

## Проблема

Сейчас ошибка источника или декодирования может приводить к генерации
синтетических пятен. Пользователь получает визуально правдоподобную картинку, но
она не является наблюдением. Для метеорологической системы это критический риск.

## Цель

Сделать происхождение каждого кадра явным и запретить использование
синтетических данных в operational-пути.

## Проектное решение

Ввести объект результата загрузки кадра или последовательности:

```text
RadarFrame
  data
  timestamp_utc
  station
  source
  product
  status
  qc_flags
  provenance
```

Допустимые статусы:

- `observed`: реальные декодированные данные;
- `missing`: данные отсутствуют;
- `decode_failed`: файл получен, но не декодирован;
- `source_failed`: источник недоступен;
- `demo`: явно запрошенные синтетические данные.

## Задачи

1. Убрать скрытые вызовы `_generate_fallback_grid()` из live AWS, FTP и local
   operational-веток.
2. Вынести demo-генератор в отдельный адаптер `DemoRadarAdapter`.
3. Запретить подготовку production-датасета при наличии `demo`,
   `decode_failed` или неизвестного provenance.
4. Добавить в API статус источника, количество реальных кадров и список ошибок.
5. На UI показывать заметный баннер `DEMO`, если пользователь явно включил demo.
6. Не запускать инференс, если минимально допустимое число наблюдений не набрано.
7. Добавить журнал качества ingestion с кодами ошибок.

## Тесты

- Ошибка AWS возвращает `source_failed`, а не массив.
- Ошибка декодирования возвращает `decode_failed`.
- Demo-адаптер работает только при явном выборе.
- Dataset builder отклоняет demo-последовательность.
- UI различает ошибку, отсутствие данных и demo.

## Риски

- После запрета fallback часть привычных сценариев перестанет визуально
  работать. Это ожидаемый результат: ложная картинка хуже явной ошибки.
- Потребуется изменить сигнатуры адаптеров и API.

## Критерий завершения

Ни одна ошибка источника или декодирования не приводит к выдаче синтетического
прогноза без явной маркировки пользователя.
