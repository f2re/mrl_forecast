# Этап 10. План практической реализации

## Цель

Перевести проект от демонстрационного ConvLSTM-контура к воспроизводимой системе подготовки радиолокационных данных и экспериментального прогноза радиоэха. Приоритет — работающий результат без избыточной архитектуры.

Основные принципы реализации:

- **KISS**: минимум обязательных сущностей и зависимостей;
- **DRY**: CLI и веб-интерфейс используют одну прикладную логику;
- **SOLID**: источник данных, декодирование, приведение к сетке, датасет и модель не смешиваются в одном классе;
- небольшое число проверок на критические контракты вместо большого набора хрупких тестов;
- изменения выполняются последовательно в `main`.

## Зафиксированные решения

1. Целевой унифицированный растр: `512 x 512`, разрешение `1 км`, локальная проекция AEQD, радиус покрытия около `256 км`.
2. Исходные файлы хранятся без изменения; унифицированный кадр является производным продуктом.
3. `нет эха` и `нет данных` всегда разделяются через `valid_mask` и `coverage_mask`.
4. Архив хранит фактические сроки наблюдений. `mtime` не считается штатным временем МРЛ.
5. Основной вход модели: не менее `60 минут` истории.
6. Для российских ДМРЛ основной целевой профиль: `6` входных и `6` выходных сроков с шагом `10 минут`.
7. Текущий профиль `4 x 15 минут` сохраняется как совместимый baseline до завершения миграции.
8. Новая модель обязана отдельно описывать перенос и эволюцию радиоэха:
   - `MotionHead` — поле перемещения;
   - `GrowthHead` — усиление;
   - `DecayHead` — ослабление;
   - `UncertaintyHead` — неопределённость.
9. Обучение запускается из терминала и из интерфейса через единый job runner.
10. Интерфейс перерабатывается в адаптивном визуальном стиле macOS/iOS без копирования фирменных элементов Apple.

## Источники данных

### Количественные источники для обучения

- NOAA NEXRAD Level II — текущий референсный источник;
- DWD Open Data — следующий открытый источник для проверки переносимости;
- ODIM HDF5/BUFR/NetCDF — через форматные адаптеры после появления реальных fixtures;
- российские ДМРЛ-BUFR — только после получения проверенного открытого файла или открытого endpoint.

### Источники для обнаружения и визуального контроля

- WIS2 Global Discovery Catalogue — поиск открытых машинно-читаемых наборов;
- Meteoinfo — визуальный композит, не количественный `dBZ`;
- RainViewer — краткий визуальный/оперативный источник с отдельной проверкой условий и семантики.

Источник не получает статус `training_allowed`, пока не подтверждены единицы, геометрия, время наблюдения, маска покрытия и условия использования.

## Этапы

### P0. Контракт данных и адаптеры — в работе

Результат этапа:

- простой модуль единого формата кадра;
- описание возможностей источника;
- реестр адаптеров;
- формат последовательности `.npz` с отражаемостью и маской;
- masked loss и masked evaluation для текущего baseline;
- сохранение обратной совместимости с legacy `.npy`.

Критерий готовности: один и тот же canonical frame используется подготовкой датасета и может быть передан визуализации без потери масок и provenance.

### P1. Открытые источники и каталог

- перенести NOAA AWS на новый контракт;
- добавить WIS2 discovery client;
- добавить Meteoinfo как `visualization_only`;
- добавить DWD как количественный открытый источник;
- реализовать мониторинг появления открытых российских радарных наборов;
- построить SQLite-каталог файлов, сроков, станций и QC.

Критерий готовности: интерфейс и CLI показывают доступность, формат и назначение каждого источника.

### P2. Поиск событий и формирование выборки

- классифицировать последовательности как `dry_valid`, `weak_echo`, `precipitation`, `convective`, `severe_core`, `invalid`;
- считать площадь порогов `5/20/30/40 dBZ`, максимум, покрытие и изменение площади;
- сбалансировать только train sampler;
- validation/test сохраняют естественное распределение;
- split выполнять по событиям и временным блокам до построения скользящих окон.

Критерий готовности: датасет воспроизводимо строится из каталога и содержит отчёт о балансе классов.

### P3. Сильные baseline и верификация

- persistence;
- global shift;
- local block motion или optical flow;
- masked MAE/RMSE;
- CSI/POD/FAR и frequency bias;
- FSS по нескольким масштабам;
- ошибка площади и максимальной отражаемости по lead time.

Критерий готовности: нейросеть не публикуется, если не превосходит baseline на целевых сроках и порогах.

### P4. `MRL-PhysEvolution`

Минимальная архитектура:

```text
reflectivity + valid_mask + range/beam features
              |
              v
       lightweight encoder
              |
              v
          ConvGRU core
          /    |     \
   Motion   Growth   Decay
          \    |     /
       differentiable advection
              |
              v
      reflectivity forecast
```

Первый продукт — только `0–60 минут`. Рекуррентная часть работает на уменьшенном пространственном разрешении, чтобы CPU-инференс оставался реалистичным.

Критерий готовности: доступны карты переноса, роста и распада; модель проходит quality gate против persistence и optical flow.

### P5. Job runner, CLI и интерфейс

- единый CLI `mrl` для ingest, prepare, dataset, train, evaluate и serve;
- SQLite jobs table и один worker без обязательных Redis/Celery;
- запуск, отмена, журналы и восстановление заданий;
- адаптивные экраны: обзор, источники, архив, конструктор датасета, обучение, модели, прогноз, диагностика;
- на карте доступны слои отражаемости, движения, роста, распада, масок и неопределённости.

Критерий готовности: одна и та же операция запускается из CLI и UI и создаёт одинаковые артефакты.

### P6. CPU deployment

- ONNX export;
- ONNX Runtime CPU;
- benchmark latency/RAM;
- квантизация только после проверки качества;
- отдельный inference dependency set;
- production WSGI и запрет произвольной загрузки checkpoint по пути.

Критерий готовности: прогноз на одном радаре выполняется на CPU с зафиксированной p50/p95 задержкой и ограничением RAM.

## Текущий рабочий срез

Первым кодовым срезом реализуются:

1. единый canonical contract и capabilities источников;
2. минимальный реестр адаптеров;
3. сохранение `reflectivity + valid_mask` в `.npz`;
4. masked loss для текущей ConvLSTM;
5. WIS2 discovery и явная маркировка визуальных источников.

После этого выполняется миграция NOAA-адаптера и построение каталога событий. Новую нейросеть до завершения этого среза не добавлять: EvolutionHead не должен обучаться на смешанных нулях, пропусках и несовместимых интерполяторах.
