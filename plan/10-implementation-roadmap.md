# Этап 10. План практической реализации

## Цель

Перевести проект от демонстрационного ConvLSTM-контура к воспроизводимой системе подготовки радиолокационных данных и экспериментального прогноза радиоэха. Приоритет — работающий результат без избыточной архитектуры.

Основные принципы реализации:

- **KISS**: минимум обязательных сущностей и зависимостей;
- **DRY**: CLI и веб-интерфейс используют одну прикладную логику;
- **SOLID**: источник данных, декодирование, приведение к сетке, датасет и модель не смешиваются в одном классе;
- небольшое число проверок на критические контракты вместо большого набора хрупких тестов;
- изменения выполняются последовательно в `main`.

## Зафиксированные решения

1. Целевой унифицированный растр: `512 x 512`, разрешение `1 км`, локальная проекция AEQD, радиус покрытия около `256 км`.
2. Исходные файлы хранятся без изменения; унифицированный кадр является производным продуктом.
3. `нет эха` и `нет данных` всегда разделяются через `valid_mask` и `coverage_mask`.
4. Архив хранит фактические сроки наблюдений. `mtime` не считается штатным временем МРЛ.
5. Основной вход модели: не менее `60 минут` истории.
6. Для российских ДМРЛ основной целевой профиль: `6` входных и `6` выходных сроков с шагом `10 минут`.
7. Текущий профиль `4 x 15 минут` сохраняется как совместимый baseline до завершения миграции.
8. Новая модель отдельно описывает перенос и эволюцию радиоэха:
   - `MotionHead` — поле перемещения;
   - `GrowthHead` — усиление;
   - `DecayHead` — ослабление;
   - `UncertaintyHead` — неопределённость.
9. Обучение запускается из терминала и из интерфейса через единый job runner.
10. Интерфейс реализуется в адаптивном визуальном стиле macOS/iOS без копирования фирменных элементов Apple.

## Источники данных

### Количественные источники

- NOAA NEXRAD Level II — референсный источник, legacy и canonical adapters реализованы;
- DWD Open Data — ODIM HDF5 DBZH adapter реализован, перед включением станции в train обязателен `check_dwd_source.py --decode-one`;
- ODIM HDF5/BUFR/NetCDF — расширяются по реальным fixtures;
- российские ДМРЛ-BUFR — только после получения проверенного открытого файла или открытого endpoint.

### Источники обнаружения и визуального контроля

- WIS2 Global Discovery Catalogue — поиск открытых машинно-читаемых наборов;
- Meteoinfo — визуальный композит, не количественный `dBZ`;
- RainViewer — краткий визуальный/оперативный источник с отдельной проверкой условий и семантики.

Источник не получает рабочее разрешение на обучение, пока не подтверждены единицы, геометрия, время наблюдения, маска покрытия и условия использования.

## Этапы

### P0. Контракт данных и адаптеры — реализовано

Реализовано:

- единый canonical contract;
- capabilities и реестр источников;
- сетка `512 x 512 / 1 км`;
- `.npz` с отражаемостью, маской и сроками;
- masked loss и masked evaluation;
- обратная совместимость с legacy `.npy`;
- запрет `mtime` как штатного срока наблюдения.

Следующий долг P0: протащить дополнительные `coverage/clutter/interpolation` masks в export и пользовательскую диагностику для всех источников.

### P1. Открытые источники и каталог — частично реализовано

Реализовано:

- NOAA AWS legacy и canonical adapters;
- WIS2 discovery client;
- Meteoinfo и RainViewer как visual-only;
- DWD quantitative ODIM HDF5 adapter;
- health-checks для NOAA, DWD и открытых discovery-источников.

Следующие задачи:

1. добавить DWD raw archive downloader и штатный dataset job;
2. сохранять контрольные суммы исходных файлов;
3. построить простой SQLite-каталог файлов, сроков, станций и QC;
4. продолжить мониторинг WIS2 на предмет открытых российских ДМРЛ.

### P2. Поиск событий и формирование выборки — базово реализовано

Реализовано:

- классы `dry_valid`, `weak_echo`, `precipitation`, `convective`, `severe_core`, `invalid`;
- площади порогов и статистика последовательностей;
- 50/50 dry/echo sampling только для train;
- естественное распределение validation;
- temporal split с удалением пересечения окон.

Следующий долг: event/date-block split до построения окон и отдельный неизменяемый test set.

### P3. Baseline и верификация — частично реализовано

Реализовано:

- persistence;
- global shift advection;
- masked MSE;
- CSI/POD/FAR по срокам и порогам;
- uniform-field gate.

Следующие задачи:

1. local block motion или optical flow;
2. FSS по нескольким масштабам;
3. frequency bias и ETS;
4. ошибка площади и максимальной отражаемости по lead time.

### P4. `MRL-PhysEvolution` — рабочая реализация

Архитектура содержит:

```text
reflectivity + valid_mask
              |
              v
       lightweight encoder
              |
              v
          ConvGRU core
       /      |      \
  Motion    Growth   Decay
       \      |      /
     differentiable advection
              |
              v
      reflectivity forecast
```

Дополнительно реализован `UncertaintyHead`, physics-guided loss и диагностические карты. Первый рабочий горизонт — `0–60 минут`.

Следующие задачи: обучение на реальном canonical archive, сравнение с optical flow и CPU benchmark.

### P5. Job runner, CLI и интерфейс — рабочая реализация

Реализовано:

- общий model runtime для веба и терминала;
- SQLite jobs table и один worker;
- запуск download/prepare/train из UI;
- терминальный prepare/train/inference;
- отмена, журналы и восстановление статусов;
- адаптивный интерфейс с тёмной темой;
- переключение слоёв отражаемости, переноса, роста, распада и неопределённости;
- выбор `MRL-PhysEvolution` или ConvLSTM из интерфейса.

Следующий долг: объединить команды в единый исполняемый CLI `mrl` и добавить source-specific ingest jobs.

### P6. CPU deployment — к выполнению

- ONNX export;
- ONNX Runtime CPU;
- benchmark latency/RAM;
- квантизация только после проверки качества;
- отдельный inference dependency set;
- production WSGI и запрет произвольной загрузки checkpoint по пути.

Критерий готовности: прогноз одного радиолокатора выполняется на CPU с зафиксированной p50/p95 задержкой и ограничением RAM.

## Ближайший рабочий срез

1. DWD raw archive downloader и подключение к dataset job.
2. SQLite-каталог исходных кадров и QC.
3. Local block-motion/optical-flow baseline.
4. Event-based split и независимый test set.
5. Обучение `MRL-PhysEvolution` на canonical masked dataset и CPU benchmark.
6. Продолжение поиска открытого российского DMRL/BUFR endpoint через WIS2; без фиктивной подмены визуальными тайлами.
